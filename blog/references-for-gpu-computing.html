CUDA Toolkit & Compatible Driver Version

CUDA は GPUデバイスを管理するための API Driver API と スレッドを管理するための API Runtime API という、2つのレベルの API で構成されている（Driver API は Runtime API の土台となっている低レイヤのAPIである）.

どのversion以降の driver を入れるべきかは、1) Driver API と 互換性のある HWか 2) どの CUDA Toolkit version を使いたいか によって決定する.

1) HWアーキテクチャと Driver Version の対応状況 は以下の表にまとめられている.　例えばAmpere は 450.36.06+ 以降、Turing は 410.48+ 以降である必要がある

CUDA Compatibility
CUDA Compatibility document describes the use of new CUDA toolkit components on systems with older base installations. The CUDA Toolkit is transitioning to a faster release cadence to deliver new features, performance improvements, and critical bug fixes.

2) CUDA Toolkit と Driver Version の対応状況は以下の表にまとめられている.

GPG key と cuda-keyring について

NVIDIA が配布する package (含driver）は他の Linux package と同様 GPG (GNU Privacy Guard) という暗号化ソフトウェアが利用されており，このための鍵が GPG key である．この鍵を管理する apt-keyコマンドが 2022/Q2に廃止が決定した．

第675回　apt-keyはなぜ廃止予定となったのか | gihyo.jp
サードパーティのAPTパッケージリポジトリを追加する際に使用する「apt-key」コマンドは、2020年8月の2.1.8から「廃止予定」となり、2022年の半ばには削除される予定になりました。今回はその理由と、代替手段について解説しましょう。

CUDA package も これまで apt-key によって manual で公開鍵を登録( apt-key adv ... ) する方式から，公開鍵を自動で管理する cuda-keyring package を用いる方法に変更されている．

GPG key も 04/27 に更新したのでそれ以前に local に登録された CUDA repo の GPG key は全て expired になっている．

Updating the CUDA Linux GPG Repository Key | NVIDIA Technical Blog
NVIDIA is updating and rotating the signing keys used by apt, dnf/yum, and zypper package managers beginning April 27, 2022.

apt-key 方式からの移行方法は以下．

  • 古GPG key 消す sudo apt-key del 7fa2af80
  • 古cuda repo 消す sudo rm -rf /etc/apt/sources.list.d/cuda*
  • cuda-keyring 入れる arch=$(/usr/bin/arch) distribution=$(. /etc/os-release;echo $ID$VERSION_ID | sed -e s/\\.//) && sudo curl -sL -o/var/cache/apt/archives/cuda-keyring_1.0-1_all.deb https://developer.download.nvidia.com/compute/cuda/repos/$distribution/$arch/cuda-keyring_1.0-1_all.deb && sudo dpkg -i /var/cache/apt/archives/cuda-keyring_1.0-1_all.deb
    • ↑ /etc/apt/sources.list.d/cuda-ubuntu1804-x86_64.list と usr/share/keyrings/cuda-archive-keyring.gpg と /etc/apt/preference.d/cuda-repository-pin-600 が勝手に作成される
  • sudo apt update

Pre-installation Actions

後述 する a. runfile / b. package manager どちらからでもインストールできるが、runfile 由来の driver と package manager 由来 の driver が共存しないよう、事前に uninstall しておく.

但し, nvidia-* を全て消すと nvidia-container 系も消えてしまうため、残したい場合は個別に uninstall する.

$ dpkg -l | grep nvidia
$ sudo apt purge nvidia-*

runfile 由来の driver は package manager からは見えないため、 sudo ./NVIDIA-Linux-x86-XXX.XX.run --uninstall で対応する.

driver 本体は以下 a~c いずれか でインストールできるが、a を推奨

sudo 無しで docker を実行したい場合 $ sudo gpasswd -a $USER docker のように docker グループに対象ユーザを追加してやれば良い.

docker image の保存先を /var/lib/docker から変更したい場合は /lib/systemd/system/docker.service において、適当な領域を -g /path/to/dir に指定してやれば良い.

# /lib/systemd/system/docker.service
- ExecStart=/usr/bin/dockerd -H fd:// --containerd=/run/containerd/containerd.sock
+ ExecStart=/usr/bin/dockerd -g /hdd1/docker -H fd:// --containerd=/run/containerd/containerd.sock
$ sudo service docker stop
$ sudo systemctl daemon-reload
$ sudo service docker start

Compute Capabilities

HWバージョンを表すため NVIDIA では Compute Capability (CC) という用語を使用している. major 番号が同一の場合、コアアーキテクチャは同一. -gencode arch=compute_50,code=sm_50 のように nvcc からアーキテクチャを指定してコードを生成する際にも利用する. （ compute_ は PTX 生成時に使われ, sm_ は PTX→バイナリ生成時に使われる）

CUDA プラットフォームで想定しているのは上方互換性であるため、旧世代アーキテクチャ向けに生成したコードは新世代アーキテクチャ上でも動作するが、新世代向けに生成したコードは旧世代アーキテクチャで動かすことはできない. -gencode を利用すれば異なるアーキテクチャ毎に最適なコードを生成し、1つのバイナリに集約することができる.
